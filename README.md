产品文档管理规范

一、 文档管理目标
云产品的文档，从多个产品对比来看，AWS是严谨、标杆的厂商，因此得出结论：
 
文档的导航、内容布局，跟AWS看齐；
部分AWS没有的产品，文档的导航、内容布局可参考阿里云。
 
二、 内容规范
1.       文档上线时间约定：产品上线时，务必同时上线产品介绍页和配套产品文档
2.       文档的组成部分：产品介绍页、文档中心、产品概述页、快速入门、产品文档
 
额外解释一下：
文档中心：指的是文档的入口页：https://www.qcloud.com/document/product
产品概述页：指的是进入具体某个产品前的那页，对应的aws截图如下

 
快速入门：本身在产品文档里有，但是也可单独提取出来，放在其他地方，比如文档中心，或者概述页里面
 
三、 流程管理
A.         新文档上线流程
1.         各团队产品经理首先将外部各链接的描述、文档概述页的内容、入门教程、文档、《产品文档竞品对比表.xlsx》，5大类文档提交给各中心文档接口人；文档接口人请参见第六条
2.         文档接口人审核：需要确保上述5大类资料是齐全的，不全的应予打回；文档接口人请额外参照本规范第四条进行审核
3.         产品leader审核：文档接口人审核通过后，将文档提交给产品leader；产品leader请额外参照本规范第四条进行审核
a.    外部各链接的描述：产品介绍页中需要加链接；该产品在文档中心的位置，排序是否合理，在文档中心的产品名称是否和官网导航一致；
b.    文档概述页的内容：内容布局与AWS对齐；AWS没有的产品，至少应提供用户指南、API参考、SDK参考（可选）、快速入门；
c.     入门教程：是否满足快速上手的原则；是否和文档中的快速入门保持一致
d.    文档本身：如果AWS有对应产品，那么目录结构与AWS看齐；如果AWS没有对应产品，目录参照《产品文档竞品对比表.xlsx》的一、二级目录；竞品对比表中给出了详细对比的，觉得阿里云更合理的可参照阿里云的模式，leader要仔细评估这里的差异点；
e.    竞品对比表：格式请参考《产品文档竞品对比表.xlsx》
4.         Frankzheng审核：Leader审核通过后，请文档接口人将上述5大类资料提交给frankzheng进行审核；如果发现问题较多，会直接打回，不会整体看完；frankzheng请额外参照本规范第四条进行审核
5.         发布：Frankzheng审核通过后，中心文档接口人安排走发布流程
 
B.         变更流程
1.         各团队产品经理将修改项提交中心文档接口人
2.         文档接口人审核：文档接口人请额外参照本规范第四条进行审核
3.         产品leader审核：中心文档接口人将修改项提交给产品leader，产品leader根据《产品文档竞品对比表.xlsx》结构来审核文档内容；产品leader请额外参照本规范第四条进行审核
4.         发布：产品leader审核通过后，中心文档接口人安排走发布流程
 
四、注意事项
A.         文档的组成：根据第二条第2点，检查提交的材料是否有缺失
B.         错别字：错别字要特别注意，各级审核人员要识别；
C.         目录缺失：《产品文档竞品对比表.xlsx》中的非可选项，都是必填项，不应该缺失目录；对象模块的识别，个别产品之前出现一些偷懒的情况，模块介绍不全，各级审核人员要识别；
D.        产品名称一致性：产品名称要跟官网首页导航保持一致，不可随意书写；
E.         内容顺序：各类列表中的排序，需要符合惯例，不可随意排列；产品排放，应跟官网首页导航顺序保持一致；目录排放应该跟竞品对比表保持一致；
F.          命名合理性：概念命名，要通俗易懂，最好不要有歧义；另外，不要把内部的概念随意暴露给客户，客户往往理解不了；

## 2 文档说明

### 2.1 文档更新目录

允许更新的SVN目录

* trunk/docs/demo/static

* trunk/docs/demo/_posts

新建目录，请使用英文目录名。

static文件夹存放静态文件，比如图片，引用路径../static/images/test/test.png。 建议使用压缩工具[TinyPNG](http://www.tinypng.com), 将图片压缩再引用。 理论上是可以将其他文件，比如视频，pdf等文件放入其中，但是为了减小服务器带宽压力，建议使用CDN存放，文档中引用链接。

\_posts文件夹存放文档内容，仅md文件。_posts中的内容将全部被发布到线上，请不要将无关文件提交到\_posts文件。草稿请放在trunk/docs/demo/\_data目录下。

### 2.2 文档格式
发布文档，请使用Markdown格式文件。

#### 2.2.1 文件命名

文件命名格式:[year]-[month]-[day]-[englishName].md

* year 四位数字，比如1000，建议按照既有顺延

* month 两位有效月份，比如01

* day 两位有效日期，比如02，建议仅取值01-28

* englishName 英文文档名，比如bkDevenglishName，不要使用index,about等敏感词汇

#### 2.2.2 文件ID说明
比如文件：1234-23-21-testName.md

文件的ID为：342321

文件ID是一个六位数字：年份的后两位+月+日

**请确保文件ID唯一。**

#### 2.2.3 文档的显示顺序
为了能通过文件名，直观地看到排序结果。 **文档顺序是文件名中日期的逆序**，比如1000年在1001年前面。

如果有需要调整文档的顺序，可以通过调整年份的前两位大小。

#### 2.2.4 文档内容
文档的前几行一定要定义meta，比如：

* layout: post

* title: 蓝鲸文档发布指引

* category: bkDev

* keywords: FAQ

* tag: [FAQ,bkMan]

其中，

layout的值一定要为post。

category可选项：
* bkProd，产品文档

* bkDev，新手入门

* bkESB，ESB文档

* bkAPI，API文档

* bkRule，开发规范

* bkTips，流程指引

* bkSolu，解决方案

* bkRes，资源下载

* bkFAQ，FAQ

如果category属于新手入门，ESB文档，API文档之一，请在tag中新增bkMan，将其增加到【开发指南】
